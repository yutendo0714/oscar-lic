# G4 Go/No-Go Decision

**Date:** 2026-06-25  
**Scope:** frozen-base OSCAR-LIC enhancement feasibility after MLIC++ corrected checkpoint, actual-byte `.oscr` accounting, multi-teacher OCR, true latent candidate labels and Eval100 selector diagnostics.

## Decision

**Conditional Go** for the OSCAR-LIC frozen-base enhancement research direction.

**No-Go** for these specific simple variants:

- raw int8 latent residual patches as the final transmitted enhancement;
- residual-magnitude-only candidate selection;
- residual-MSE decoder-known codebook as the final compact payload;
- wide flat latent-vector tabular selectors at Eval100 scale.

The central hypothesis remains alive because true counterfactual latent candidate utility exists and survives actual-byte accounting. The next proposal should be a utility-aware gate plus compact utility-trained enhancement symbols.

## Evidence

### Actual-Byte Base Path

The corrected MLIC++ checkpoint is locked and evaluated through actual `compress/decompress` and `.oscr` files, not likelihood-estimated bitrate. The checkpoint is:

- `experiments/checkpoints/mlicpp/mlicpp_mse_0025_corrected.pth.tar`
- SHA256 `67af8c950a4e8ae03da9bc95b87d13fd7831063ad12b7df1f872154e0082c559`

The base-only `.oscr` wrapper decodes correctly, and optional enhancement sections can be skipped for base-compatible decode.

### Oracle Utility

Eval100 true single-candidate latent residual labels show real OCR headroom:

| Evaluator | MLIC++ base CER/exact | Best single-candidate oracle CER/exact |
|---|---:|---:|
| PARSeq | 0.0791 / 0.8000 | 0.0640 / 0.8500 |
| Tesseract | 0.4030 / 0.3600 | 0.3427 / 0.3800 |

On 20 image-group splits, the oracle selects about 6.75/25 validation images at 4.0503 actual bpp and improves both recognizers:

| Policy | Actual bpp | PARSeq CER/exact | Tesseract CER/exact |
|---|---:|---:|---:|
| Base | 3.4257 | 0.0846 / 0.7960 | 0.4111 / 0.3640 |
| Oracle single candidate | 4.0503 | 0.0658 / 0.8560 | 0.3444 / 0.3820 |

This passes the minimum feasibility criterion: selected latent modifications can improve task utility under complete transmitted-byte accounting.

### Non-Oracle Selectors

Selectors are not strong enough yet, but they do learn signal:

| Selector | Main finding |
|---|---|
| Linear latent/stat selector | Eval100 val AUC 0.7085 on one split, but actual one-candidate policy mostly misses oracle gains. |
| Per-image MLP ranker | First non-oracle selector to consistently reduce Tesseract CER, but only modestly. |
| Image-statistics ranker | Val AUC about 0.67/AP 0.18; improves over residual RMS but does not close the oracle gap. |
| Wide latent-vector ranker | Overfits and worsens actual policy. |
| Patch-CNN ranker | Val AUC 0.7470/AP 0.3127, but policy-level OCR gains remain modest. |

This means row-wise candidate classification is easier than policy-level selection. Future validation must prioritize actual-bpp OCR policy metrics rather than AUC alone.

### Payload Diagnostics

Raw latent residual payloads are too expensive:

| Text24 method | Actual bpp | Enhancement payload bpp | OCR finding |
|---|---:|---:|---|
| Raw residual 5% | 4.8040 | 1.6117 | Tesseract CER recovers to original-crop level, PARSeq unchanged. |
| Raw residual 10% | 5.8902 | 2.6979 | Higher PSNR but no Tesseract gain. |

Codebook payloads are compact but not utility-aligned:

| Eval100 method | Actual bpp | Enhancement payload bpp | PARSeq CER/exact | Tesseract CER/exact |
|---|---:|---:|---:|---:|
| MLIC++ base | 3.4641 | 0.0000 | 0.0791 / 0.8000 | 0.4030 / 0.3600 |
| K16 residual-MSE codebook 5% | 3.8495 | 0.0865 | 0.0791 / 0.8000 | 0.4030 / 0.3500 |
| Oracle-selected K16 codebook | 3.5637 | 0.0223 | 0.0734 / 0.8200 | 0.3766 / 0.3600 |
| Oracle-selected K64 -> 26 centers | 3.5637 | 0.0223 | 0.0659 / 0.8400 | 0.3917 / 0.3600 |

This supports a compact codebook/symbol stream, but only if paired with utility-aware selection and eventually trained with OCR/task utility. The oracle-selected K16 result is especially important: the payload is tiny, and both PARSeq and Tesseract improve over MLIC++ base.

K64 nearly reaches the PARSeq raw oracle but trades away much of the Tesseract gain, so symbol capacity and training objective must be multi-teacher-aware rather than simply more faithful to residual vectors.

## Next Required Step

Proceed with a compact, utility-trained enhancement prototype:

1. Keep the MLIC++ base frozen and preserve base-only decoding.
2. Use binary multi-teacher improvement plus pairwise/listwise ranking for the gate target.
3. Train a small shared local encoder for candidate selection, but validate by policy-level OCR and actual total bpp.
4. Train codebook or low-dimensional residual symbols with utility-aware objectives, using the oracle-selected K16 result as the first positive compact-payload target.
5. Expand labels beyond Eval100 before claiming model-level performance.

The proposal axis should be stated as:

> OSCAR-LIC adds an optional task-utility enhancement layer over a strong frozen learned codec, using actual-byte-aware candidate gating and compact utility-trained enhancement symbols.

This axis preserves the novelty while avoiding the dead ends identified by the current diagnostics.
